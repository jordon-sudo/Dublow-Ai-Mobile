v0.2.5 - Workflows, Jobs list, etc.
Added
- src/lib/appsTypes.ts - single source of truth for app/workflow shapes. Exports AppItem, WorkflowApp, the isWorkflow type guard (narrows to WorkflowApp), and isTerminalStatus to centralize the pending | running | complete | failed state machine. All downstream modules import from here; no more scattered string literals.
- listAppsRaw / getAppRaw on the API client - return server-shaped AppItem (with user_inputs, prompt_sections/steps, variable_type) for the new tabbed Apps screen and dynamic form renderer. Coexist with the legacy listApps / getApp (which still flatten to AppInfo via normalizeApp) so existing callers are untouched. Legacy pair to be removed once all screens migrate.
- listApps name filter - accepts a name query forwarded to the server. The Apps screen's search input is debounced (300 ms) and filters server-side; no client-side fuzzy matching.
- workflowJobsStore (encrypted, persisted) - single source of truth for workflow runs, consumed by the Jobs screen, foreground poller, and future background-poll/notification layer.
    - WorkflowJob holds the raw server snapshot; TrackedJob wraps it with local metadata (start time, notified flag, submitted inputs) so server shape changes cannot corrupt local bookkeeping.
    - updateJob(id, snapshot) returns true only on the non-terminal -> terminal transition - the precise hook for firing a notification. Idiom: if (await updateJob(id, snap)) { fireNotification(id); await markNotified(id); }. Two-step commit prevents duplicate notifications under poller races.
    - notified is persisted to disk (not in-memory) so a killed-and-relaunched app does not re-notify completed jobs.
    - Pruning on hydrate: terminal jobs older than 7 days dropped; total list capped at 100.
    - Ordering is creation-time (most recent first) and stable across status changes - rows do not jump while polling.

v0.2.7 - Workflows, Jobs list, etc.
Added
- Workflow runner screen - same AppInputForm; submits to /workflows/run, receives job_id, calls trackJob before router.replace('/jobs/{job_id}') so a cold-start deep link finds the job already in local state. Uses router.replace (not push) so Back from the Jobs screen returns to the Apps list. Shows a steps-count chip (app.steps?.length) as a "this will take a moment" affordance. running is not reset on success - only on error - to avoid a flicker before navigation.
- Workflow job detail screen (app/jobs/[jobId].tsx) - live poller with terminal-state output rendering.
    - pollingRef is a mutable ref (not state) so the setInterval callback always reads the latest flag - avoids stale-closure bugs.
    - Flips to no-op on terminal status; no unnecessary network calls.
    - Every snapshot is written through updateJob so the store - not in-memory state - arbitrates the terminal transition.
    - Presigned file URLs are minted on demand against /v1/workflows/presigned-url and handed to Linking.openURL, ensuring a fresh 1-hour TTL each tap.
    - Permissive output rendering: string -> text; object -> pretty-printed JSON; file output_type -> "Open file" button.
- Jobs list screen (app/jobs/index.tsx) - lists every tracked job from workflowJobsStore, ordered most-recent-first, with status color coding and tap-through to detail.
- Jobs entry point in Apps header - clock icon (top-right of app/apps.tsx) routes to /jobs. The Jobs screen is now reachable via (a) automatic router.replace after a workflow run, (b) the new header icon, and (c) direct deep link to /jobs or /jobs/{id}.
Changed
- client.uploadFile - second argument is now an optional scope object. Existing chat-composer call site client.uploadFile({ uri, name, type }) compiles and behaves identically. When called as client.uploadFile(file, { scopeType: 'workflow', scopeId: appId }), two extra multipart fields are appended and the server treats the upload as workflow-scoped. Backward compatible.
Unchanged (intentional)
- streamChat - <thinking> sentinel parsing, SSE reader loop, tolerant event-type matching, partial-chunk handling, [DONE] sentinel, and non-JSON line tolerance are all preserved. Do not refactor; the corner cases are handled and working.

v0.2.8 - Local notifications for workflow jobs
Added
- src/lib/notifications.ts - local-notification layer built on expo-notifications.
    - ensureNotificationPermission() - idempotent permission request; honors iOS canAskAgain and guards a re-prompt on Android via a module-level flag. Creates the Android workflow-jobs channel on first grant (required for API 26+).
    - notifyJobComplete(job) - fires an immediate local notification on terminal status, with distinct title/body for complete vs failed, and data: { job_id } payload for deep-linking.
    - installNotificationTapHandler() - registers a response listener that routes taps to /jobs/{job_id} via expo-router. 50 ms deferral ensures the router is mounted when the app is cold-started from a tap.
    - Foreground setNotificationHandler set at module top level so banners appear even when the app is open.
Changed
- app/_layout.tsx - root effect now calls ensureNotificationPermission() once at startup and installs the tap handler, returning the unsubscribe on teardown.

v0.2.9 - Global foreground job poller
Added
- src/lib/jobPoller.ts - idempotent foreground poller that drives every notification.
    - 5 s interval over non-terminal jobs in workflowJobsStore; concurrency capped at 4 to avoid bursts.
    - inFlight reentrancy guard prevents overlapping ticks on slow networks.
    - AppState listener fires an immediate tick on return to foreground, catching jobs that completed while the device was locked.
    - Terminal-transition detection is store-sourced (wasTerminal read before updateJob), so it cooperates correctly with the detail screen's 3 s poller without duplicate notifications. markNotified is the second layer of defense.
    - Transient errors are logged at warn and swallowed; the next tick retries.
- workflowJobsStore.markNotified(job_id) action and TrackedJob.notified?: boolean field.
Changed
- app/_layout.tsx - startJobPoller() added inside the startup effect.
Known limitation
- Fully suspended or force-quit apps do not tick. Background execution is deferred to a future expo-background-task pass, which requires a dev build.
