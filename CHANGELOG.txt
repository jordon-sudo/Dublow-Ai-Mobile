v0.12.0 - Workflows, Jobs list, etc.
Added
- src/lib/appsTypes.ts - single source of truth for app/workflow shapes. Exports AppItem, WorkflowApp, the isWorkflow type guard (narrows to WorkflowApp), and isTerminalStatus to centralize the pending | running | complete | failed state machine. All downstream modules import from here; no more scattered string literals.
- listAppsRaw / getAppRaw on the API client - return server-shaped AppItem (with user_inputs, prompt_sections/steps, variable_type) for the new tabbed Apps screen and dynamic form renderer. Coexist with the legacy listApps / getApp (which still flatten to AppInfo via normalizeApp) so existing callers are untouched. Legacy pair to be removed once all screens migrate.
- listApps name filter - accepts a name query forwarded to the server. The Apps screen's search input is debounced (300 ms) and filters server-side; no client-side fuzzy matching.
- workflowJobsStore (encrypted, persisted) - single source of truth for workflow runs, consumed by the Jobs screen, foreground poller, and future background-poll/notification layer.
    - WorkflowJob holds the raw server snapshot; TrackedJob wraps it with local metadata (start time, notified flag, submitted inputs) so server shape changes cannot corrupt local bookkeeping.
    - updateJob(id, snapshot) returns true only on the non-terminal -> terminal transition - the precise hook for firing a notification. Idiom: if (await updateJob(id, snap)) { fireNotification(id); await markNotified(id); }. Two-step commit prevents duplicate notifications under poller races.
    - notified is persisted to disk (not in-memory) so a killed-and-relaunched app does not re-notify completed jobs.
    - Pruning on hydrate: terminal jobs older than 7 days dropped; total list capped at 100.
    - Ordering is creation-time (most recent first) and stable across status changes - rows do not jump while polling.

v0.14.0 - Workflows, Jobs list, etc.

v0.16.0 - Global foreground job poller
Added
- src/lib/jobPoller.ts - idempotent foreground poller that drives every notification.
    - 5 s interval over non-terminal jobs in workflowJobsStore; concurrency capped at 4 to avoid bursts.
    - inFlight reentrancy guard prevents overlapping ticks on slow networks.
    - AppState listener fires an immediate tick on return to foreground, catching jobs that completed while the device was locked.
    - Terminal-transition detection is store-sourced (wasTerminal read before updateJob), so it cooperates correctly with the detail screen's 3 s poller without duplicate notifications. markNotified is the second layer of defense.
    - Transient errors are logged at warn and swallowed; the next tick retries.
- workflowJobsStore.markNotified(job_id) action and TrackedJob.notified?: boolean field.
Changed
- app/_layout.tsx - startJobPoller() added inside the startup effect.
Known limitation
- Fully suspended or force-quit apps do not tick. Background execution is deferred to a future expo-background-task pass, which requires a dev build.
